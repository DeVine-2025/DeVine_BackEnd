# Flyway DB 마이그레이션 가이드

## 개요

DB 스키마 변경은 JPA `ddl-auto`가 아닌 **Flyway 마이그레이션 스크립트**로 관리합니다.

- 운영/개발: `ddl-auto: validate` (Flyway가 스키마 관리, Hibernate는 검증만)
- 테스트: `ddl-auto: none` (Flyway가 Testcontainers PostgreSQL에 스키마 생성)

## 마이그레이션 파일 작성

### 위치

```
src/main/resources/db/migration/
```

### 네이밍 규칙

```
V{타임스탬프}__{이슈번호}_{설명}.sql
```

- 타임스탬프: `YYYYMMDDHHmmSS` (커밋 시점 기준, 14자리)
- 언더스코어 **2개** (`__`)
- 이슈번호: GitHub 이슈 번호 (없으면 생략)
- 설명: snake_case

```
V20260320170000__123_add_project_deadline_column.sql
V20260321100000__create_chat_table.sql
```

### 작성 예시

**컬럼 추가**
```sql
ALTER TABLE project ADD COLUMN deadline timestamp;
```

**테이블 생성**
```sql
CREATE TABLE chat_message (
    chat_message_id bigint NOT NULL GENERATED BY DEFAULT AS IDENTITY,
    created_at timestamp(6) NOT NULL,
    content text NOT NULL,
    member_id bigint NOT NULL,
    CONSTRAINT chat_message_pkey PRIMARY KEY (chat_message_id),
    CONSTRAINT fk_chat_message_member FOREIGN KEY (member_id) REFERENCES member(member_id)
);
```

**enum 컬럼 추가 시 CHECK 제약조건 포함**
```sql
ALTER TABLE chat_message ADD COLUMN status varchar(20) NOT NULL DEFAULT 'ACTIVE';
ALTER TABLE chat_message ADD CONSTRAINT chat_message_status_check
    CHECK (status IN ('ACTIVE', 'DELETED'));
```

## 주의사항

1. **이미 적용된 마이그레이션 파일은 절대 수정하지 마세요.** 체크섬 불일치로 앱 시작이 실패합니다.
2. **엔티티 변경 시 반드시 마이그레이션도 함께 작성하세요.** `validate` 모드에서 불일치 시 시작 실패합니다.
3. **PostgreSQL 문법으로 작성하세요.** (GENERATED BY DEFAULT AS IDENTITY, CHECK, partial index 등)
4. **enum 컬럼에는 CHECK 제약조건을 추가하세요.**
5. **partial unique index**가 필요한 경우 `CREATE UNIQUE INDEX ... WHERE ...` 사용:
   ```sql
   CREATE UNIQUE INDEX idx_name ON table (col1, col2) WHERE condition;
   ```

## 신규 로컬 환경 세팅

`baseline-version: 0`이므로 신규 환경에서 앱을 시작하면 V1부터 모든 마이그레이션이 자동 적용됩니다.

1. 앱 시작 → Flyway가 V1 ~ V3 순차 실행
2. 별도의 수동 SQL 실행 불필요

## 로컬 트러블슈팅

**체크섬 오류 발생 시** (로컬 DB에서만):
```sql
DELETE FROM flyway_schema_history WHERE version = '문제버전';
```

**처음부터 다시 시작하고 싶을 때** (로컬 DB에서만):
```sql
DROP TABLE flyway_schema_history;
-- 앱 재시작하면 baseline-on-migrate로 자동 처리
```
